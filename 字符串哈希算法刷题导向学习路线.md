# 字符串哈希到底解决什么问题？

字符串哈希的核心思想是：

```text
把字符串映射成一个数字，
然后通过比较数字来近似判断两个字符串是否相等。
```

它常用于：

- 判断两个字符串是否相等；
- 快速比较两个子串是否相等；
- 统计不同字符串数量；
- 统计不同子串数量；
- 判断回文；
- 找最长重复子串；
- 找最长公共子串；
- 配合二分答案；
- 替代部分 KMP / Z 函数场景；
- 优化暴力字符串比较。

需要特别注意：

```text
字符串哈希不是绝对正确算法。
```

因为不同字符串可能产生相同哈希值，这叫：

```text
哈希碰撞 / 冲突。
```

工程和竞赛中通常通过：

```text
双哈希
大模数
自然溢出 unsigned long long
随机 base
```

来降低碰撞概率。

## 多项式哈希

对于字符串：

```text
s = s[0]s[1]...s[n-1]
```

可以定义：

```text
hash(s) = s[0] * base^(n-1)
        + s[1] * base^(n-2)
        + ...
        + s[n-1]
```

实际写法中经常递推：

```cpp
long long h = 0;

for (char c : s) {
    h = (h * base + value(c)) % mod;
}
```

其中：

```text
base：进制基数，常见如 131、13331、911382323 等
mod：取模数，常见如 1e9+7、1e9+9、大质数
```

## unsigned long long 自然溢出写法

也可以不用显式取模，利用 `unsigned long long` 的自然溢出：

```cpp
using ull = unsigned long long;

ull getHash(const string& s) {
    ull h = 0;
    const ull base = 131;

    for (char c : s) {
        h = h * base + c;
    }

    return h;
}
```

这种写法速度快，但仍然可能碰撞。

## 本阶段常见错误

- 以为哈希值相等就一定是字符串相等；
- 字符值从 0 开始，导致某些字符串哈希不稳定；
- 忘记大小写敏感；
- base 太小或 mod 太小；
- 使用 int 存哈希导致溢出；
- 没有考虑空字符串；
- 把“哈希表 map/unordered_map”和“字符串哈希”混为一谈。

| 4 | 洛谷 P2957 [USACO09OCT] Barn Echoes G | https://www.luogu.com.cn/problem/P2957 | 找两个字符串的最长重叠部分，可用哈希比较前后缀 |

## 前缀哈希模板

使用 1 下标更容易写子串哈希：

```cpp
using ull = unsigned long long;

const ull base = 131;
const int N = 1000005;

ull h[N], p[N];

void build(const string& s) {
    int n = s.size();
    p[0] = 1;

    for (int i = 1; i <= n; i++) {
        h[i] = h[i - 1] * base + s[i - 1];
        p[i] = p[i - 1] * base;
    }
}

ull getHash(int l, int r) {
    // l, r 为 1 下标，表示 s[l...r]
    return h[r] - h[l - 1] * p[r - l + 1];
}
```

## 为什么能 O(1) 求子串哈希？

如果：

```text
h[r] = s[1] * base^(r-1) + ... + s[r]
h[l-1] = s[1] * base^(l-2) + ... + s[l-1]
```

那么把 `h[l-1]` 乘上：

```text
base^(r-l+1)
```

就能和 `h[r]` 对齐。

所以：

```text
hash(l, r) = h[r] - h[l-1] * base^(r-l+1)
```

## 判断回文

可以同时维护：

```text
原串前缀哈希
反串前缀哈希
```

要判断 `s[l...r]` 是否为回文，就比较：

```text
s[l...r] 的哈希
reverse(s)[n-r+1 ... n-l+1] 的哈希
```

如果相等，就认为该子串是回文。

## 本阶段常见错误

- 子串下标从 0 开始和 1 开始混用；
- `p[r-l+1]` 的长度写错；
- 使用有符号 long long 自然溢出导致未定义风险；
- 双哈希时只比较其中一个哈希值；
- 反串位置映射错误；
- 模数哈希中出现负数没有加 mod；
- 没有预处理足够长的幂数组。

## 最长重复子串

目标：

```text
找到字符串中出现至少两次的最长子串。
```

二分答案：

```text
如果存在长度为 mid 的重复子串，
那么长度小于 mid 的重复子串也一定存在。
```

因此可以二分长度。

检查函数：

```text
1. 枚举所有长度为 len 的子串；
2. 计算每个子串哈希；
3. 放入 set；
4. 如果某个哈希出现过，说明可能存在重复子串。
```

伪代码：

```cpp
bool check(int len) {
    unordered_set<ull> seen;

    for (int l = 1; l + len - 1 <= n; l++) {
        int r = l + len - 1;
        ull cur = getHash(l, r);

        if (seen.count(cur)) {
            return true;
        }

        seen.insert(cur);
    }

    return false;
}
```

## 多字符串最长公共子串

可以对长度 `len` 做二分。

检查时：

```text
1. 对第一个字符串所有长度为 len 的子串哈希放入集合；
2. 对第二个字符串取交集；
3. 对第三个字符串继续取交集；
4. 如果最后集合非空，说明存在公共子串。
```

这类题需要特别注意：

```text
哈希碰撞风险和集合去重。
```

如果数据强，建议使用双哈希。

## 本阶段常见错误

- 二分边界写错；
- check 函数没有清空 set；
- 哈希碰撞导致误判；
- 只用哈希不验证原串，结果被卡；
- 多字符串公共子串中没有做集合交集；
- unordered_set 被卡常数，可以改用 sort 后去重；
- 返回具体子串时没有记录位置。

## 双哈希

单哈希可能碰撞，因此常用双哈希：

```cpp
struct Hash {
    long long x, y;

    bool operator==(const Hash& other) const {
        return x == other.x && y == other.y;
    }

    bool operator<(const Hash& other) const {
        if (x != other.x) return x < other.x;
        return y < other.y;
    }
};
```

常见模数：

```text
mod1 = 1e9 + 7
mod2 = 1e9 + 9
```

或者：

```text
unsigned long long + 一个大质数 mod
```

双哈希思想：

```text
只有两个哈希值都相等，才认为字符串相等。
```

## 子串周期判断

要判断字符串 `s[l...r]` 是否存在长度为 `p` 的周期，可以比较：

```text
s[l ... r-p] == s[l+p ... r]
```

用哈希可以 O(1) 判断这两个子串是否相等。

如果：

```text
len = r - l + 1
len % p == 0
且 s[l ... r-p] == s[l+p ... r]
```

则 `p` 是这个子串的周期。

P3538 就是这类方向的代表题。

## 哈希 + LCP 二分

如果要比较两个后缀从某个位置开始的最长公共前缀长度，可以二分长度 `mid`：

```text
如果 hash(a, a+mid-1) == hash(b, b+mid-1)，说明 LCP 至少为 mid。
```

这种方法可以在没有后缀数组的情况下，快速实现一些 LCP 查询。

## 统计不同子串

最直接方法：

```text
枚举长度 len；
把所有长度为 len 的子串哈希放入 set；
累加不同哈希数量。
```

复杂度：

```text
O(n^2 log n)
```

适合 n 不太大的情况。

如果 n 很大，应该考虑：

```text
后缀数组
后缀自动机
```

## 本阶段常见错误

- 复杂题仍只用单哈希，被卡碰撞；
- 判断周期时忘记 `len % p == 0`；
- LCP 二分时越界；
- 统计不同子串时复杂度过高；
- unordered_set 存 pair 没有自定义哈希；
- 回文判断时反串映射错误；
- 哈希比较代替字典序比较时，没有先求 LCP。

# 每道题的复盘模板

建议每做完一道字符串哈希题，都写下面几个问题。

## 1. 本题为什么能用字符串哈希？

例如：

```text
本题需要频繁比较两个子串是否相等，直接比较会超时，因此用前缀哈希实现 O(1) 子串比较。
```

## 2. 哈希对象是什么？

例如：

```text
本题哈希的是整个字符串。
```

或者：

```text
本题哈希的是所有长度为 mid 的子串。
```

或者：

```text
本题同时维护原串和反串哈希，用于判断回文。
```

## 3. 是否需要前缀哈希？

如果要频繁查询子串哈希，通常需要：

```text
h[i]：前 i 个字符的哈希
p[i]：base 的 i 次幂
```

## 4. 是否需要双哈希？

如果题目数据较强，或会大量比较哈希，建议使用双哈希。

例如：

```text
最长重复子串
多字符串公共子串
周期判断
不同子串统计
```

## 5. 是否需要二分答案？

如果问题具有单调性：

```text
存在长度为 L 的重复子串
=> 一定存在长度小于 L 的重复子串
```

就可以考虑：

```text
二分长度 + 哈希判重
```

## 6. 是否有更确定的算法？

如果题目不能接受概率错误，或者数据特别强，可能需要：

```text
KMP
Z 函数
后缀数组
后缀自动机
Manacher
```

# 字符串哈希与其他字符串算法的选择

## 更适合字符串哈希的情况

- 需要频繁比较子串是否相等；
- 需要快速判断回文；
- 需要二分 LCP；
- 需要二分答案判断重复子串；
- 题目允许使用概率型方法；
- 想用较短代码解决复杂字符串比较。

## 更适合 KMP 的情况

- 单模式串匹配；
- 需要确定性算法；
- 需要前后缀结构；
- 需要构造 KMP 自动机。

## 更适合 Z 函数的情况

- 需要每个后缀与整串前缀的 LCP；
- 需要统计字符串得分；
- 需要判断大量后缀是否等于前缀。

## 更适合 Manacher 的情况

- 题目核心是回文子串；
- 需要所有中心的回文半径；
- 需要 O(n) 统计回文。

## 更适合后缀数组 / 后缀自动机的情况

- 需要严格统计不同子串数量；
- 需要处理大量 LCP 查询；
- 需要字典序相关后缀问题；
- 数据规模大，哈希 O(n^2) 不可接受。

# 最终掌握标准

学完字符串哈希后，看到一道题应该能快速判断：

1. 是否需要频繁比较字符串或子串？
2. 是整串哈希，还是前缀哈希？
3. 子串哈希公式是否写对？
4. 是否需要反串哈希判断回文？
5. 是否需要二分答案？
6. 是否需要双哈希降低碰撞风险？
7. 是否需要记录具体位置，而不只是判断存在？
8. 是否可能被哈希碰撞卡掉？
9. 是否应该改用 KMP、Z 函数、Manacher、后缀数组或后缀自动机？

如果这些问题都能回答出来，说明你已经能把字符串哈希作为一种快速子串比较工具，而不是只会写 `h = h * base + c`。
